3. Deployment — Managing Pods

You almost never create Pods directly. A Deployment manages a ReplicaSet which manages Pods.

Deployment → ReplicaSet → Pods (Animated Chain)
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app       # ← must match template labels!
  template:
    metadata:
      labels:
        app: my-app     # ← label that Service will find
    spec:
      containers:
      - name: my-app
        image: my-app:1.0
        ports:
        - containerPort: 8080

4. Service & Ports — The Part That Confuses Everyone

This is your #1 confusion. Watch the animated traffic flow through each port type:

Animated Port Mapping — Watch the Traffic!
Port NameWho Uses ItExampleAnalogy
portThe Service itself80The front door of the Service
targetPortThe Pod/Container8080The actual room where your app runs
nodePortExternal world → Node30080The street address anyone can visit
containerPortDeployment spec (docs)8080A label saying what the app uses
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-svc
spec:
  type: NodePort
  selector:
    app: my-app           # ← MUST match Pod labels!
  ports:
  - port: 80              # ← Service listens here
    targetPort: 8080      # ← forwards to container's 8080
    nodePort: 30080       # ← external access (30000-32767)
The Golden Rule
targetPort in Service MUST equal containerPort in Deployment. The port can be anything.

6.4 The imagePullPolicy Explained

PolicyWhen Does K8s Pull?Default ForUse When
AlwaysEvery time a pod startstag = latest or no tagDev environments where you reuse tags
IfNotPresentOnly if image not in node cacheAny specific tag (v1.0, abc123)Production with unique tags (default & best)
NeverNever pulls — must be pre-loadedNothingAir-gapped environments only
The "latest" Tag Trap
When you use :latest, K8s sets imagePullPolicy: Always by default. BUT if the YAML doesn't change, kubectl apply won't trigger a new rollout at all — so the "Always" pull policy never gets a chance to run! That's why rollout restart is needed.

6.5 ConfigMap/Secret Changes — Same Problem!

Updating a ConfigMap or Secret also does NOT restart pods automatically:

# You update a ConfigMap:
kubectl edit configmap app-config    # change some values

# Pods still use OLD config! Need to restart:
kubectl rollout restart deployment/my-app

# PRO TIP: Use a hash annotation to auto-trigger updates!
spec:
  template:
    metadata:
      annotations:
        configHash: "{{ sha256sum of configmap }}"   # changes when config changes!
        # Helm can do this automatically with:
        # checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
Your Action Plan — Stop Deleting Deployments!
Right now: Use kubectl rollout restart deployment/my-app instead of delete + apply. Zero downtime.
This week: Switch from :latest to unique tags like :v1.0.1 or :$(git rev-parse --short HEAD)
This month: Set up a CI/CD pipeline that auto-tags images. Never touch kubectl manually again.

7. Volumes & Mounts — Where Data Lives

Animated Volume Mount Flow

Volume Types

TypeUse CaseSurvives Restart?Survives Delete?
emptyDirTemp scratch, share between containers✅❌
hostPathMount host directory✅✅ (same node)
persistentVolumeClaimDatabase, stateful apps✅✅
configMapConfig files✅N/A
secretPasswords, certs✅N/A
# Two-step process: 1) Define volume, 2) Mount it
spec:
  containers:
  - name: my-app
    volumeMounts:                    # STEP 2: WHERE to mount
    - name: data-vol               # ← references by name
      mountPath: /data
    - name: config-vol
      mountPath: /config
      readOnly: true
  volumes:                            # STEP 1: WHAT to mount
  - name: data-vol
    persistentVolumeClaim:
      claimName: data-pvc
  - name: config-vol
    configMap:
      name: app-config
Mental Model
1) Define the volume (in spec.volumes) → 2) Mount it (in volumeMounts). The name connects them — like plugging a USB drive into a folder.

11. Networking Deep Dive — How Services Talk to Deployments

This is the most confusing part of Kubernetes, so let's go layer by layer. Every Pod gets its own IP address, and every Service gets a stable virtual IP (ClusterIP). Here's exactly how traffic flows.

11.1 The Full Request Lifecycle — Step by Step

Animated: Complete Request Journey (External → Pod)
Key Insight — Service is NOT a Process!
A Service doesn't run anywhere. It's a set of iptables/IPVS rules on every node, maintained by kube-proxy. When traffic hits the ClusterIP, the kernel's netfilter does a DNAT (Destination NAT) to a random healthy Pod IP. That's why Services are so fast — it's kernel-level networking, not a proxy.

11.2 The Label → Selector Connection (The Glue!)

This is the #1 thing people miss. The Service finds Pods through label matching, not by name or deployment reference.

Animated: How Service Discovers Pods via Labels
Common Mistake #1 — Labels Don't Match!
If your Service selector says app: my-app but your Deployment labels say app: myapp (no hyphen), the Service will find zero Endpoints and all requests get 503 errors. Always verify with: kubectl get endpoints my-svc

11.5 Service Discovery — Two Methods

Method 1: DNS (Recommended)

# In your app code or env var:
DB_HOST=db-svc.backend

# Full FQDN:
DB_HOST=db-svc.backend.svc.cluster.local

# Same namespace (shortest):
API_URL=http://api-svc:80/v1

CoreDNS resolves service names to ClusterIPs. This is what you should use 99% of the time.

Method 2: Environment Variables

# K8s auto-injects for every Service:
echo $API_SVC_SERVICE_HOST
# → 10.96.45.12

echo $API_SVC_SERVICE_PORT
# → 80

# Naming: {SVC_NAME}_SERVICE_HOST
# (uppercase, dashes become underscores)

Only works for Services created BEFORE the Pod. DNS is more reliable.

11.6 NetworkPolicy — The Firewall

By default, all pods can talk to all other pods. NetworkPolicy adds firewall rules to restrict traffic.

# Only allow frontend namespace to reach backend db-svc
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-db
  namespace: backend
spec:
  podSelector:
    matchLabels:
      app: postgres          # ← apply to DB pods
  policyTypes: [Ingress]
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: frontend      # ← only from frontend ns
    ports:
    - port: 5432
Debugging Networking — The Commands You Need
kubectl get endpoints api-svc — are pods registered? If empty, labels don't match!
kubectl describe svc api-svc — check selector, ports, and endpoints
kubectl exec -it debug-pod -- nslookup api-svc — does DNS resolve?
kubectl exec -it debug-pod -- curl api-svc:80/health — can you actually reach it?
kubectl get networkpolicy -A — any policies blocking traffic?

12. Ingress — The Smart Front Door

Ingress sits at the edge of your cluster and routes external HTTP/HTTPS traffic to the right Service based on URL path or hostname.

Animated Ingress Routing — Watch Traffic Split by URL Path
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  tls:
  - hosts: [app.example.com]
    secretName: tls-secret          # ← TLS cert stored as a Secret
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-svc            # ← which Service to route to
            port:
              number: 80              # ← the Service's port (not targetPort)

18. Common Kubernetes Pain Points & Troubleshooting

Every K8s engineer hits these walls. Here's every common pain point organized by category, with the exact kubectl commands to diagnose and fix each one. Click any category to expand.

🚀 Deployment & Update Issues 7 pain points ▼
VERY COMMON kubectl apply doesn't update pods
Why: Image tag (e.g. :latest) didn't change → K8s sees identical YAML → no rollout triggered. Node uses cached image.
Fix:
# Quick fix — force restart (zero downtime):
kubectl rollout restart deployment/my-app

# Best fix — use unique image tags:
image: my-app:v1.0.2   # or git commit SHA

# Check current image:
kubectl get deploy my-app -o jsonpath='{.spec.template.spec.containers[0].image}'
VERY COMMON CrashLoopBackOff
Why: Container starts, crashes, K8s restarts it, crashes again → exponential backoff (10s, 20s, 40s...). Usually: app error, missing env var, wrong command, or config file not found.
Diagnose:
kubectl logs my-pod                     # see crash output
kubectl logs my-pod --previous           # logs from LAST crashed container
kubectl describe pod my-pod              # check Events section at bottom
kubectl get events --sort-by=.lastTimestamp   # cluster-wide events
VERY COMMON ImagePullBackOff / ErrImagePull
Why: K8s can't pull the image. Wrong image name/tag, private registry without credentials, or network issue.
Diagnose:
kubectl describe pod my-pod | grep -A5 Events
# Look for: "Failed to pull image" with the exact error

# Fix: Wrong image name → correct it in deployment.yaml
# Fix: Private registry → create imagePullSecret:
kubectl create secret docker-registry regcred \
  --docker-server=registry.example.com \
  --docker-username=user --docker-password=pass

# Then add to deployment:
spec:
  imagePullSecrets:
  - name: regcred
COMMON ConfigMap/Secret changes not applied to pods
Why: Updating a ConfigMap/Secret does NOT restart pods. Existing pods keep the old values.
Fix:
kubectl rollout restart deployment/my-app    # restart pods to pick up changes

# Pro tip: use a hash annotation in Helm to auto-trigger:
annotations:
  checksum/config: {{ include "mychart/configmap.yaml" . | sha256sum }}
COMMON Rolling update stuck — new pods not becoming Ready
Why: New pods fail readinessProbe → K8s won't kill old pods → update hangs. Could be wrong health endpoint, port mismatch, or app takes too long to start.
Diagnose:
kubectl rollout status deployment/my-app     # see rollout progress
kubectl get pods                              # new pods stuck at 0/1 Running
kubectl describe pod my-app-NEW-POD           # check readiness probe failures
kubectl rollout undo deployment/my-app        # ROLLBACK if broken!
COMMON Pod stuck in Pending state
Why: No node has enough CPU/memory, or node selector/affinity/taint doesn't match any node.
Diagnose:
kubectl describe pod my-pod | grep -A10 Events
# Look for: "Insufficient cpu" or "Insufficient memory" or "no nodes match"

kubectl top nodes                            # check node resource usage
kubectl describe node NODE_NAME              # see allocatable vs allocated
OCCASIONAL CreateContainerConfigError
Why: Pod references a ConfigMap, Secret, or ServiceAccount that doesn't exist.
Diagnose:
kubectl describe pod my-pod
# Look for: "configmap 'xxx' not found" or "secret 'xxx' not found"
kubectl get configmap,secret -n my-namespace  # check if they exist
🌐 Networking & Service Issues 6 pain points ▼
VERY COMMON Service returns "connection refused" — 0 Endpoints
Why: Service selector labels don't match Pod labels. Service finds no pods → empty Endpoints → all traffic fails.
Diagnose & Fix:
kubectl get endpoints my-svc               # EMPTY = labels don't match!
kubectl describe svc my-svc                 # check Selector field
kubectl get pods --show-labels              # compare with selector

# The selector in Service MUST match labels in Deployment template:
# Service:    selector: app=my-app
# Deployment: labels:   app=my-app  ← must be IDENTICAL
VERY COMMON Port confusion — targetPort vs port vs containerPort
Why: targetPort in Service must equal the port your app actually listens on (containerPort). If these don't match → connection refused.
Check:
# What port does your app listen on?
kubectl exec my-pod -- netstat -tlnp        # or ss -tlnp
# If app listens on 8080, then Service targetPort MUST be 8080

# Quick test — port-forward directly to pod:
kubectl port-forward pod/my-pod 8080:8080   # bypass Service to test
COMMON DNS not resolving cross-namespace service
Why: From namespace A, you can't just call svc-name in namespace B. You must include the namespace: svc-name.namespace-b
Fix:
# Wrong (from frontend ns trying to reach backend ns):
curl http://db-svc:5432           # ✗ looks in current ns only!

# Correct:
curl http://db-svc.backend:5432   # ✓ includes namespace

# Debug DNS:
kubectl exec -it my-pod -- nslookup db-svc.backend
kubectl exec -it my-pod -- cat /etc/resolv.conf
COMMON Ingress not routing / 404 errors
Why: Ingress controller not installed, wrong annotations for your controller type, or backend service name/port mismatch.
Diagnose:
kubectl get ingress                          # check ADDRESS column
kubectl describe ingress my-ingress          # check rules and backend
kubectl get pods -n ingress-nginx            # is controller running?
kubectl logs -n ingress-nginx CONTROLLER_POD # check for errors
COMMON Pods can't reach external internet
Why: NetworkPolicy blocking egress, DNS issues (CoreDNS not working), or cloud NAT gateway not configured.
Diagnose:
kubectl exec -it my-pod -- nslookup google.com    # DNS works?
kubectl exec -it my-pod -- curl -v https://api.example.com
kubectl get networkpolicy -A                       # any egress policies?
OCCASIONAL Service works on some nodes but not others
Why: kube-proxy not running on all nodes, or iptables rules out of sync. Can also be CNI plugin issues.
Diagnose:
kubectl get pods -n kube-system -l k8s-app=kube-proxy  # running on all nodes?
kubectl logs -n kube-system kube-proxy-XXXXX
💾 Storage & Volume Issues 5 pain points ▼
VERY COMMON PVC stuck in Pending
Why: No PV matches the PVC (wrong StorageClass, capacity too large, or access mode mismatch). Or the StorageClass provisioner is not installed.
Diagnose:
kubectl describe pvc my-pvc                  # check Events for reason
kubectl get pv                                # any Available PVs?
kubectl get storageclass                      # is the StorageClass available?

# Common fix — check StorageClass name matches:
# PVC: storageClassName: "gp2"
# SC:  name: "gp2"              ← must match!
COMMON Data lost when pod moves to different node
Why: Using hostPath volume — data only exists on one node. If pod is rescheduled to another node, data is gone.
Fix: Use PVC with a network-attached storage (AWS EBS, GCP PD, NFS, Ceph). Never use hostPath in production for important data.
COMMON Volume mount hides existing container files
Why: Mounting a volume to /app replaces everything in /app from the image. Your app binary disappears!
Fix: Mount to a subdirectory like /app/data instead. Or use subPath to mount a single file without hiding others.
volumeMounts:
- name: config
  mountPath: /app/config/settings.yaml
  subPath: settings.yaml       # ← only mounts this one file!
COMMON ReadWriteOnce blocks pod scheduling
Why: RWO volume can only be mounted by pods on ONE node. If you scale replicas, pods on other nodes can't attach the same volume.
Fix: Use ReadWriteMany (NFS, EFS) for shared access, or use StatefulSet with per-pod PVCs, or design your app to not need shared storage.
DANGEROUS PV data deleted when PVC is removed
Why: Default reclaimPolicy: Delete destroys the PV (and data!) when PVC is deleted.
Fix:
# Set reclaimPolicy to Retain for important data:
kubectl patch pv my-pv -p '{"spec":{"persistentVolumeReclaimPolicy":"Retain"}}'

# Or set it in StorageClass:
reclaimPolicy: Retain
📊 Resource & Performance Issues 4 pain points ▼
VERY COMMON OOMKilled — Out of Memory
Why: Container exceeded its memory limit. K8s kills it immediately. Often: Java apps with high heap, memory leaks, or limit set too low.
Diagnose & Fix:
kubectl describe pod my-pod | grep -i oom
kubectl top pod my-pod                        # current memory usage

# Increase memory limit:
resources:
  limits:
    memory: "512Mi"   # was "256Mi" — increase it
  requests:
    memory: "256Mi"   # guaranteed minimum
COMMON App is slow — CPU throttling
Why: CPU limit is too low. K8s throttles the container when it tries to use more. Unlike memory, it doesn't kill — it just slows down.
Diagnose:
kubectl top pod my-pod                        # current CPU usage
# If CPU is near the limit → throttled → increase limit
# Pro tip: Some teams remove CPU limits entirely and only set requests
COMMON No resource limits → noisy neighbor problem
Why: One pod without limits consumes all CPU/memory on a node → other pods starve. Causes cascading failures.
Fix: ALWAYS set requests and limits. Use LimitRange to enforce defaults per namespace.
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
spec:
  limits:
  - default:        # default limits if not specified
      cpu: "500m"
      memory: "256Mi"
    defaultRequest: # default requests
      cpu: "100m"
      memory: "128Mi"
    type: Container
OCCASIONAL Evicted pods — node under pressure
Why: Node runs low on disk, memory, or PIDs. kubelet evicts pods starting with those that exceed requests. BestEffort pods (no requests) get evicted first.
Diagnose:
kubectl describe node NODE | grep -A5 Conditions
kubectl get pods --field-selector=status.phase=Failed  # see evicted pods
🔒 RBAC & Security Issues 4 pain points ▼
VERY COMMON Forbidden — RBAC permission denied
Why: Your user or ServiceAccount doesn't have the right Role/ClusterRole bound to it.
Diagnose & Fix:
# Check what you CAN do:
kubectl auth can-i get pods                   # yes/no
kubectl auth can-i create deployments         # yes/no
kubectl auth can-i --list                     # list all permissions

# Check as a ServiceAccount:
kubectl auth can-i get pods --as=system:serviceaccount:my-ns:my-sa
COMMON Pod blocked by PodSecurity admission
Why: Namespace has PodSecurity Standards enforced (restricted/baseline). Your pod tries to run as root or use privileged settings.
Fix: Add securityContext to run as non-root:
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
DANGEROUS Secrets committed to Git in plain text
Why: Base64 is NOT encryption. Anyone who reads the YAML can decode it. Committing secrets to Git = security breach.
Fix: Use Sealed Secrets (Bitnami), External Secrets Operator, or HashiCorp Vault. Never commit plain Secrets to Git.
CONFUSING Role vs ClusterRole confusion
Why: Role = namespace-scoped (only works in one namespace). ClusterRole = cluster-wide. Using Role when you need ClusterRole = "Forbidden" on resources in other namespaces.
Rule of thumb: Use Role + RoleBinding for namespace-specific access. Use ClusterRole + ClusterRoleBinding for cluster-wide access (nodes, PVs, namespaces).
⎈ Helm & GitOps Issues 4 pain points ▼
COMMON Helm values override confusion
Why: Multiple sources of values: chart defaults → values.yaml → -f custom.yaml → --set flags. Later ones override earlier. Easy to lose track.
Debug:
helm get values my-release                   # see current values
helm get values my-release --all             # including defaults
helm template my-release ./chart -f vals.yaml # preview rendered YAML
COMMON Helm upgrade fails — can't rollback
Fix:
helm history my-release                      # see all revisions
helm rollback my-release 3                   # rollback to revision 3
helm upgrade --atomic my-release ./chart      # auto-rollback on failure!
COMMON ArgoCD stuck "OutOfSync"
Why: Live cluster state differs from Git. Could be a manual kubectl edit, auto-generated fields (like service.spec.clusterIP), or Helm hook resources.
Fix: Click "App Diff" in ArgoCD UI to see exactly what differs. Add ignoreDifferences in ArgoCD for auto-generated fields. Avoid manual kubectl edits — everything goes through Git.
OCCASIONAL Helm chart version not pinned — breaking updates
Fix: Always pin chart versions:
helm install my-app bitnami/nginx --version 15.3.2   # pin it!

19. Security Best Practices

  • Use Namespaces for isolation
  • Apply NetworkPolicies (zero-trust)
  • Use RBAC — never cluster-admin for apps
  • Run containers as non-root
  • Set resource limits (CPU/memory) on every container
  • Scan images with Trivy in CI
  • Encrypt Secrets at rest
  • Use Service Mesh (Istio/Linkerd) for mTLS
