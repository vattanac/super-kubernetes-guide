1. The Big Picture — How Kubernetes Works

Before diving into details, here is how every piece fits together. Think of Kubernetes as a city: Nodes are buildings, Pods are apartments, Services are phone numbers, and Volumes are storage rooms.

Kubernetes Architecture Overview

2. Pod — The Smallest Unit

A Pod is the smallest deployable unit in Kubernetes. It wraps one or more containers that share the same network (localhost) and storage.

Inside a Pod
Key Insight
Containers inside the same Pod can talk to each other via localhost. They also share the same IP address. This is why a Pod is not the same as a container — it's a group of tightly-coupled containers.

3. Deployment — Managing Pods

You almost never create Pods directly. Instead, you create a Deployment which manages a ReplicaSet, which manages the Pods. This gives you rolling updates, rollbacks, and scaling.

Deployment → ReplicaSet → Pods
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app       # ← must match template labels!
  template:                   # ← this IS the Pod spec
    metadata:
      labels:
        app: my-app     # ← label that Service will find
    spec:
      containers:
      - name: my-app
        image: my-app:1.0
        ports:
        - containerPort: 8080  # ← port your app listens on

4. Service & Ports — The Part That Confuses Everyone

This is your #1 confusion, so let's break it down carefully. A Service is a stable network endpoint that routes traffic to Pods. Pods come and go (they get new IPs), but a Service stays at the same address.

The 3 Ports You Must Understand

Port Mapping: How Traffic Flows
Port NameWho Uses ItExampleThink of it as...
portThe Service itself80The front door of the Service
targetPortThe Pod/Container8080The actual room inside where your app runs
nodePortExternal world → Node30080The street address anyone can visit
containerPortDeployment spec (docs only)8080A label on the door saying what port the app uses
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-svc
spec:
  type: NodePort          # ClusterIP | NodePort | LoadBalancer
  selector:
    app: my-app           # ← MUST match Pod labels from Deployment!
  ports:
  - port: 80              # ← Service listens on this
    targetPort: 8080      # ← forwards to container port 8080
    nodePort: 30080       # ← external access (30000-32767)
The Golden Rule
targetPort in the Service MUST equal containerPort in the Deployment. If your app listens on 8080, then targetPort: 8080. The port can be anything — it's just the Service's own port.

Service Types Compared

🔒

ClusterIP (default)

Only accessible inside the cluster. Other pods call it by name: my-app-svc:80. Use for internal services like databases.

🚪

NodePort

Opens a port (30000-32767) on every node. Access from outside: NodeIP:30080. Good for dev/testing.

☁️

LoadBalancer

Creates an external cloud load balancer (AWS ELB, GCP LB). Gets a public IP. The standard for production.

🗺️

ExternalName

Maps to a DNS name (e.g., my-db.rds.amazonaws.com). No proxying — just DNS aliasing.

5. Volumes & Mounts — Where Data Lives

Containers are ephemeral — when they restart, all data inside is lost. Volumes solve this by providing persistent storage that survives container restarts.

How Volume Mounting Works

Volume Types Cheatsheet

TypeUse CaseData Survives Pod Restart?Data Survives Pod Delete?
emptyDirTemp scratch space, share between containers✅ Yes❌ No
hostPathMount a directory from the host node✅ Yes✅ Yes (on same node)
persistentVolumeClaimDatabase storage, stateful apps✅ Yes✅ Yes
configMapConfig files injected into Pod✅ YesN/A (external)
secretPasswords, tokens, certs✅ YesN/A (external)
nfsShared file storage across pods✅ Yes✅ Yes
# Pod spec with volume mounts
spec:
  containers:
  - name: my-app
    image: my-app:1.0
    volumeMounts:                    # ← WHERE to mount inside container
    - name: data-vol               # ← references volume by name
      mountPath: /data             # ← path inside container
    - name: config-vol
      mountPath: /config
      readOnly: true
  volumes:                            # ← WHAT to mount (defined at Pod level)
  - name: data-vol
    persistentVolumeClaim:
      claimName: data-pvc
  - name: config-vol
    configMap:
      name: app-config
Two-Step Mental Model
Always think: 1) Define the volume (in spec.volumes) → 2) Mount it (in spec.containers[].volumeMounts). The name field connects them. Like plugging a USB drive (volume) into a specific folder (mountPath).

6. PersistentVolume (PV) & PersistentVolumeClaim (PVC)

PV / PVC Relationship — Like a Hotel Reservation
# pv.yaml — created by admin or dynamically by StorageClass
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce      # RWO=one node, ROX=many read, RWX=many read/write
  hostPath:
    path: /mnt/data
---
# pvc.yaml — created by developer
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi       # ← K8s finds a PV ≥ 5Gi and binds it

8. ConfigMap — Non-Secret Configuration

ConfigMaps store configuration data as key-value pairs. You can inject them as environment variables or as files mounted into pods.

As Environment Variables

env:
- name: DB_HOST
  valueFrom:
    configMapKeyRef:
      name: app-config
      key: database_host

As Mounted Files

volumeMounts:
- name: cfg
  mountPath: /etc/config
volumes:
- name: cfg
  configMap:
    name: app-config
# Each key becomes a file!
# Create ConfigMap from command line
kubectl create configmap app-config \
  --from-literal=database_host=postgres-svc \
  --from-literal=database_port=5432 \
  --from-file=nginx.conf=./nginx.conf

9. Secrets — Sensitive Data

Secrets are like ConfigMaps but for sensitive data (passwords, API keys, TLS certificates). They are base64 encoded (not encrypted by default!).

Security Warning
Base64 is NOT encryption! Anyone with cluster access can decode secrets. In production, enable encryption at rest and use tools like Sealed Secrets, Vault, or External Secrets Operator. 
# Create a secret
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password=supersecret123

# Use in a Pod
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-creds
      key: password

# Or mount as files
volumeMounts:
- name: secret-vol
  mountPath: /etc/secrets
  readOnly: true
volumes:
- name: secret-vol
  secret:
    secretName: db-creds
🔑

Opaque (generic)

Default type. Store any key-value pairs like usernames, passwords, API keys.

🔐

kubernetes.io/tls

TLS certificates. Requires tls.crt and tls.key fields.

🐳

kubernetes.io/dockerconfigjson

Docker registry credentials for pulling private images.

11. Ingress — Exposing to the World

Ingress is like a smart receptionist that routes external HTTP/HTTPS traffic to the right Service based on the URL path or hostname.

Ingress Routing
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  tls:
  - hosts: [app.example.com]
    secretName: tls-secret
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-svc
            port:
              number: 8080

12. DNS in Kubernetes

Every Service automatically gets a DNS entry. This is how pods find each other — by name, not IP.

Kubernetes DNS Hierarchy
my-service                                  ← same namespace (shortest)
my-service.my-namespace                      ← cross-namespace
my-service.my-namespace.svc                  ← with service type
my-service.my-namespace.svc.cluster.local    ← fully qualified (FQDN)

15. GitOps & ArgoCD — The Modern Way

GitOps means Git is the single source of truth. You push YAML changes to Git, and ArgoCD automatically syncs them to your cluster. No more kubectl apply from your laptop!

🔄

ArgoCD

The most popular GitOps tool. Watches your Git repo and auto-syncs changes to K8s. Provides a beautiful UI to see deployment status.

🔀

Flux

CNCF GitOps toolkit. Lightweight alternative to ArgoCD. Good for teams who prefer CLI over UI.

🏗️

Kustomize

Built into kubectl. Lets you customize YAML without templates. Great for managing different environments (dev/staging/prod).

16. Monitoring & Observability

📊

Prometheus + Grafana

Industry standard. Prometheus scrapes metrics, Grafana visualizes them. Use kube-prometheus-stack Helm chart for easy setup.

📝

EFK Stack

Elasticsearch + Fluentd + Kibana. Collects and searches logs from all pods. Alternative: Loki + Grafana (lighter).

🔍

Lens IDE

Free desktop app — a visual K8s dashboard. See pods, logs, metrics all in one place. Great for beginners.

🩺

Health Probes

Liveness, Readiness, Startup probes. K8s uses these to know if your app is healthy and ready to receive traffic.

# Health Probes — Add to every Deployment!
containers:
- name: my-app
  livenessProbe:       # ← Is the app alive? Restart if not.
    httpGet:
      path: /healthz
      port: 8080
    initialDelaySeconds: 15
    periodSeconds: 10
  readinessProbe:      # ← Is it ready for traffic? Remove from LB if not.
    httpGet:
      path: /ready
      port: 8080
    initialDelaySeconds: 5
    periodSeconds: 5

17. Security Best Practices

  • Use Namespaces to isolate teams/environments
  • Apply NetworkPolicies to restrict pod-to-pod traffic (zero-trust)
  • Use RBAC (Role-Based Access Control) — never give cluster-admin to apps
  • Run containers as non-root users
  • Set resource limits (CPU/memory) on every container
  • Use Pod Security Standards (restricted, baseline, privileged)
  • Scan images with Trivy or Snyk in your CI pipeline
  • Encrypt Secrets at rest (EncryptionConfiguration)
  • Use Service Mesh (Istio/Linkerd) for mTLS between pods
  • Keep K8s version updated (security patches)
# Resource limits — ALWAYS set these!
containers:
- name: my-app
  resources:
    requests:           # ← Minimum guaranteed
      cpu: "100m"      # 0.1 CPU core
      memory: "128Mi"
    limits:             # ← Maximum allowed
      cpu: "500m"
      memory: "256Mi"  # OOMKilled if exceeded!

18. Essential Tools Beyond Kubernetes

🐳

Docker / Podman

Build container images. Podman is a rootless alternative. Learn Dockerfile best practices (multi-stage builds, .dockerignore).

🏗️

Terraform / OpenTofu

Infrastructure as Code. Provision cloud resources (VPCs, EKS clusters, databases). The foundation under Kubernetes.

⚡

k3s / k3d / kind

Lightweight K8s for local dev. k3s runs on a Raspberry Pi. kind runs K8s in Docker. Perfect for learning.

🔒

HashiCorp Vault

Production-grade secrets management. Dynamic secrets, encryption as a service, PKI. Way better than K8s Secrets alone.

🌐

Istio / Linkerd

Service Mesh: automatic mTLS, traffic management, observability. Linkerd is simpler; Istio is more powerful.

📦

Harbor / ECR / GCR

Container registries. Harbor is self-hosted with vulnerability scanning. ECR/GCR are cloud-managed.

19. Speed-Learning Tips for Busy Professionals

🎯

Tip 1: Learn by Breaking

Create a cluster with kind, deploy something, then deliberately break it. Delete a pod, misconfigure a port, use wrong labels. Debugging teaches faster than reading.

📋

Tip 2: The 3-YAML Rule

Most apps need just 3 files: Deployment, Service, Ingress. Master these first. Everything else is optional until you need it.

🔧

Tip 3: kubectl explain

kubectl explain deployment.spec.template — built-in docs for any field. Better than googling. Use --recursive to see all fields.

⚡

Tip 4: Aliases Save Hours

Add to your .bashrc: alias k=kubectl, alias kgp='kubectl get pods', alias kaf='kubectl apply -f'. Speed compounds.

🧪

Tip 5: CKA Certification Path

Study for CKA even if you don't take it. The curriculum covers exactly what you need. Use killer.sh for practice.

📺

Tip 6: Visual Learner Resources

Nana TechWorld (YouTube) — best visual K8s tutorials. KodeKloud — hands-on labs. Play with Kubernetes — free browser-based clusters.

Your 30-Day Learning Plan
Week 1: Pod, Deployment, Service, Labels/Selectors (the connection glue!)
Week 2: ConfigMaps, Secrets, Volumes, PVC
Week 3: Ingress, Namespaces, RBAC, Health Probes
Week 4: Helm, CI/CD, ArgoCD, Monitoring stack

Spend 30 min/day on hands-on practice. That's enough when combined with work experience.

20. kubectl Cheatsheet

# ━━━ GET INFO ━━━
kubectl get pods                         # list pods in current namespace
kubectl get pods -A                      # ALL namespaces
kubectl get pods -o wide                 # show IPs and nodes
kubectl get svc,deploy,ingress           # multiple resources
kubectl describe pod my-pod              # detailed info + events

# ━━━ DEBUGGING ━━━
kubectl logs my-pod                      # view logs
kubectl logs my-pod -c sidecar           # specific container
kubectl logs my-pod --previous           # logs from crashed container
kubectl exec -it my-pod -- /bin/sh       # shell into pod
kubectl port-forward svc/my-svc 8080:80  # local access to service

# ━━━ CREATING / UPDATING ━━━
kubectl apply -f deployment.yaml         # create or update
kubectl delete -f deployment.yaml        # delete resources
kubectl scale deploy my-app --replicas=5 # scale up/down
kubectl rollout restart deploy my-app    # restart all pods
kubectl rollout undo deploy my-app       # rollback to previous version

# ━━━ DRY RUN (generate YAML) ━━━
kubectl create deploy my-app --image=nginx --dry-run=client -o yaml > deploy.yaml
kubectl expose deploy my-app --port=80 --dry-run=client -o yaml > svc.yaml

# ━━━ CONTEXT & NAMESPACE ━━━
kubectl config get-contexts              # list clusters
kubectl config use-context my-cluster    # switch cluster
kubectl config set-context --current --namespace=dev  # change default ns
Pro Tip: kubectx + kubens
Install kubectx and kubens for instant context and namespace switching. Way faster than typing long kubectl config commands.
